# Hot To Complete Tasks

## Общаа информация

### Где искать задачи

Задачи публикуются в Issues репозитория. Надо брать только те задачи,
которые создал автор репозитория (https://github.com/nelkor).

### Ветка

Для новой задачи актуализируем ветку `main` и создаём от неё новую.
Имя ветки начинаем с `feature/` или `bug/` в зависимости от сути задачи.
Добавляем номер Issue и в `kebab-case` краткую суть задачи.
Пример: `feature/90-connect-font`.

### Коммиты

Одна задача — один коммит. Имя коммита — "subj".

### Изменения кода

В основном, изменения кода должны касаться только текущей задачи.
Если для решения задачи будет оптимальным поправить стороннюю логику,
это также надо сделать.

Если нужно импортировать что-то из другого слайса, но он это не ре-экспортирует,
можно добавить ре-экспорт в `index.ts` того слайса.

Добавлять новые npm-библиотеки "просто так" нежелательно,
но если есть причина, то можно.

### Pull Request

Результатом выполнения задачи является Pull Request к ветке `main`.
Имя PR начинается с emoji `:sparkles:` для `feature` и `:bug:` для `bug`.
Далее в квадратных скобках номер Issue и в `Title Case` имя задачи.
Пример: `✨ [90] Connect Font`.

В описании PR следует указать:
- Сколько времени заняло выполнение задачи.
- Сколько и каких ресурсов было потрачено.
- Были ли моменты, не предвиденные условием задачи.
- Какие новые npm-библиотеки были добавлены.
- В чём заключается решение и почему оно такое.

## Алгоритм для менеджера (Сеньор Краб)

Менеджер НЕ пишет код. Менеджер координирует:
берёт задачу, готовит ветку, запускает субагента, создаёт PR.

### Шаг 1 — Найти задачу

Зайти в Issues. Найти одну открытую задачу без комментария "Беру задачу".
Кратко понять суть — достаточно для имени ветки. В детали не углубляться.
Если задач нет или все взяты — сообщить руководителю и остановиться.

### Шаг 2 — Комментарий

Написать в Issue: "Беру задачу".

### Шаг 3 — Подготовить ветку

```
git checkout main
git pull
git checkout -b feature/N-kebab-name # или bug/N-kebab-name
git push -u origin {имя ветки}
```

### Шаг 4 — Запустить субагента

- Передать задачу ДОСЛОВНО, без пересказа.
- Субагент сам читает: README.md + все файлы /docs/*.
- Субагент сам читает текст задачи из Issue.
- Субагент доводит работу до готового коммита (git add, git commit).
- В конце субагент передаёт менеджеру текст описания для PR и завершает работу.
- Параметры: runtime="subagent", runTimeoutSeconds=600.

### Шаг 5 — Создать PR

После получения описания от субагента:

```
gh pr create --title "✨ [N] Title Case" --body "..." --base main
```

### Доклад руководителю

После каждого шага кратко сообщать:
1. Увидел задачу "...".
2. Написал комментарий.
3. Актуализировал ветку main, создал ветку {имя}.
4. Вызвал субагента, жду ответа.
5. Готово, создал PR.

### При ошибке субагента

Раследовать и сообщить руководителю причину.
Самому задачу не решать. Остановиться.
