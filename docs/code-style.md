# Code Style

По большей части, стиль кода стандартизован файлами `.eslintrc` и `.prettierrc`,
а его проверка автоматизирована. Можно написать код как угодно,
но потом обязательно запустить `npm run lint:fix`.
Если ошибка не чинится автоматически — починить вручную.

Запрещается менять конфигурацию ESLint/Prettier.

Импортируем:
- Из других *слайсов* (FSD) только по алиасу `@/`.
- Внутри слайса только по относительному пути.

Основную часть файлов именуем в `kebab-case`.
Vue-компоненты именуем в `PascalCase`.

Не используем `style` внутри Vue-компонентов. Чтобы стилизовать вёрстку,
создаём файл `index.scss` в корне слайса и импортируем его в `index.ts`
в корне слайса (пример — `src/app/index.ts`).

Разгружаем `script` компонента — выносим логику в *хуки* и *библиотеки*.

Для реактивной логики используем хуки. Каждый хук экспортирует функцию
`use{Something}`, названную в зависимости от своей зоны ответственности.

Если логика небольшая, будет один файл `hook.ts` в корне слайса,
который экспортирует, например, `useMainPage`. Если логика достаточно большая,
чтобы разделить её на несколько хуков по зонам ответственности, используем
директорию `hooks` в корне слайса (сегмент `hooks`). Это нормально,
если один хук принимает аргументом значение, полученное из другого хука.

Для чистой/утилитарной логики используем библиотеки. Либо файл `lib.ts`,
либо сегмент `lib`.

Все типы слайса описываем в файле `types.ts` в корне слайса. Типы объектов
с полями задаём ключевым словом `interface`.
Типы функций, объединения литералов — ключевым словом `type`.
Никогда не используем `enum`.
